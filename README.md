# Shoria — витрина кроссовок (Vue SPA + Laravel API)

Пет‑проект интернет‑магазина: фронт на **Vue 3 (отдельное SPA)**, бэкенд на **Laravel** с **PostgreSQL** и **Redis**. Локальная разработка через **Docker Compose** — на сервер потом переносится обычным способом: те же контейнеры или отдельный деплой (см. раздел «Дорожная карта»).

## Зачем Docker, если вы в нём не сильны

- Один раз поставили [Docker Desktop](https://www.docker.com/products/docker-desktop/) (macOS) — дальше команды из этого README.
- У всех одинаковые версии PHP, расширений и Postgres — меньше «у меня работает».
- Postgres сразу, как в продакшене, без установки СУБД в систему.

## Админка «как в Битриксе» (по смыслу, не копия)

Для контент‑менеджера (баннеры, товары, новости, категории) на Laravel лучше всего подходит **[Filament](https://filamentphp.com/)** (для **Laravel 13** ставим **Filament v4**, v3 с этой версией Laravel не совместим):

- привычные **списки** с поиском, фильтрами, сортировкой;
- **формы** с валидацией, вкладками, связями «товар → категория»;
- **роли и доступ** (через пакеты к Filament или встроенные возможности);
- при необходимости позже подключим **медиабиблиотеку** (например Spatie Media Library) — удобные галереи и файлы к баннерам.

Filament — это **панель внутри Laravel** (не Vue): для админки это плюс — быстрее вводить сущности и править контент, чем писать отдельную SPA‑админку с нуля.

### Изображения в админке: URL + загрузка файла

В формах контента теперь поддерживаются оба варианта:

- вставить внешний URL картинки;
- загрузить файл с компьютера.

Реализовано для:

- баннеров (одиночное изображение),
- категорий (одиночное изображение),
- новостей (обложка + картинки в тексте через RichEditor),
- товаров (множественные изображения в repeater).

Технически: при загрузке хранится путь в `public`-диске, а API автоматически отдает рабочий URL (`/storage/...`), поэтому фронт одинаково корректно рендерит и внешние, и локально загруженные изображения.

Витрина для покупателей остаётся на **Vue** (SEO для публичных страниц доработаем: мета, SSR/prerender при необходимости — в дорожной карте).

## Быстрый старт

Из корня репозитория (`Shoria`):

### 1) Переменные для Compose

```bash
cp .env.example .env
```

При необходимости поменяйте порты в `.env` (`HTTP_PORT`, `POSTGRES_PORT`, `VITE_PORT`).
`COMPOSE_PROJECT_NAME=shoria` лучше не менять: это фиксирует имя Docker-проекта и volume БД, чтобы данные не «переезжали» в новый пустой volume.

### 2) Laravel в папке `backend/`

```bash
make init-backend
```

Скрипт создаст проект Laravel в `./backend` через официальный образ **Composer** (интернет нужен).

### 3) Настройка `backend/.env`

Откройте `backend/.env` и выставьте подключение к БД (должно совпасть с `docker-compose.yml`):

```env
DB_CONNECTION=pgsql
DB_HOST=postgres
DB_PORT=5432
DB_DATABASE=shoria
DB_USERNAME=shoria
DB_PASSWORD=shoria
```

Сгенерируйте ключ приложения (после того как контейнеры подняты — шаг 4):

```bash
make up
make backend-key
make backend-migrate
```

### 4) Filament (админка)

```bash
make filament-install
```

Дальше создаём пользователя админки (один раз):

```bash
docker compose exec app php artisan make:filament-user
```

Панель обычно доступна по адресу вида `http://localhost:8080/admin` (порт см. `HTTP_PORT` в `.env`).

### Filament: после ввода логина/пароля страница «просто обновляется»

Чаще всего сессия не цепляется к браузеру. Проверьте по порядку:

1. Открывайте админку **тем же URL**, что в `APP_URL` (в Docker по умолчанию **`http://localhost:8080`**, не голый `http://localhost` без порта, если у вас так настроено).
2. В `backend/.env` для локального **http** выставьте `SESSION_SECURE_COOKIE=false` (если там `true`, Safari/Chrome не отправят cookie сессии).
3. Очистите куки для `localhost` и попробуйте снова.
4. Перезапуск после правок: `docker compose up -d` (или `docker compose restart app`).
5. Если в консоли браузера ошибка `GET /livewire/livewire.js ... 404`, перезапустите nginx:

```bash
docker compose restart nginx
```

Это означает, что Nginx не отдал Livewire-скрипт через Laravel роутинг (в проекте уже добавлен фикс в `docker/nginx/default.conf`).

В репозитории уже заданы `APP_URL` в `docker-compose` и `trustProxies` в `backend/bootstrap/app.php` — подтяните изменения и перезапустите контейнеры.

### Новость: картинка в тексте не отображается (битая иконка)

Зафиксировали и исправили типовую причину:

- RichEditor новостей теперь сохраняет вложения в публичный диск: `public/news-content` (`fileAttachmentsDisk('public')`).
- В `backend/.env` и `backend/.env.example` по умолчанию выставлено `FILESYSTEM_DISK=public`.
- Добавлен fallback-роут `GET /storage/{path}` через `PublicStorageController`, который отдает файлы из `storage/app/public` даже если symlink `public/storage` не создан.
- В `backend/config/filesystems.php` для `local`-диска отключен `serve`, чтобы убрать конфликт маршрута `storage/{path}` и исключить 403 на вложениях редактора.
- В `docker/nginx/default.conf` добавлен отдельный `location ^~ /storage`, чтобы Nginx не перехватывал `.webp/.jpg` как «битую статику» и не отдавал `404` до Laravel.

Если у старой новости изображение уже было сохранено ссылкой на несуществующий файл, оно не восстановится автоматически — откройте новость в админке и загрузите картинку заново (после этого будет работать стабильно).

Если `composer require` всё ещё падает из‑за **security audit** Packagist (редко, на старых образах Composer), в каталоге `backend` можно временно ослабить блокировку только для локальной разработки:

```bash
docker compose exec app composer config audit.block-insecure false
```

Потом снова `make filament-install`. На продакшене так лучше не оставлять.

### 5) Vue SPA в `frontend/`

```bash
make init-frontend
```

Запуск dev‑сервера Vite (отдельный терминал):

```bash
make dev-frontend
```

Во `frontend/.env` / `frontend/.env.development` используйте `VITE_API_URL=http://localhost:8080` (или ваш `HTTP_PORT`), чтобы SPA ходила в Laravel.

### 6) CORS и авторизация API

Для SPA на другом порту уже добавлен CORS (`backend/config/cors.php`) и подключен **Laravel Sanctum** для token‑авторизации покупателей.

## Продуктовая стратегия (обновлено: 21 апреля 2026)

Приняли рабочую модель: **demo-first -> template-second**.

Фаза 1: делаем один сильный демо-магазин с полноценным UX и бизнес-логикой.  
Фаза 2: упаковываем его в подписочный шаблон (white-label), не теряя качество демо.

Цель: сначала получить реально продающий и удобный магазин, затем превратить его в продукт, который быстро запускается для новых клиентов.

## Что берем из лучших e-commerce практик (UX + продажи)

Ниже практики, которые доказанно влияют на конверсию и удобство, и которые закладываем в продукт:

- **Максимально короткий checkout** (минимум обязательных полей, явные шаги и ошибки рядом с полем).
- **Ранний и прозрачный расчет итоговой стоимости** (товар, скидка, доставка, финальная сумма до оплаты).
- **Сильная карточка товара**: понятные фото, наличие по размеру, сроки доставки, возврат/гарантии рядом с CTA.
- **Доверие в критических точках**: способы оплаты, политика возврата, контакты, юридическая информация, заметные trust-блоки.
- **Mobile-first скорость и простота**: легкие страницы, оптимизация изображений, меньше визуального шума в важных сценариях.
- **Контент как канал продаж**: новости/гайды/подборки, которые помогают SEO и повышают повторные визиты.
- **Система событий аналитики по воронке**: от просмотра карточки до оплаты, чтобы улучшать не “на вкус”, а по цифрам.

Референс-направления по структуре и UX-паттернам (для нас это ориентиры, не копирование дизайна 1-в-1):

- Amazon / Ozon: ясная структура карточки, сильный фокус на наличии, доставке и доверии.
- Zalando / Lamoda: удобный выбор размеров и вариативность фильтров каталога.
- Nike / Apple Store: визуальная чистота, четкие CTA и аккуратная подача бренда.

## Шаблонная архитектура (для продажи по подписке)

Слои продукта:

1. **Core (ядро)** — каталог, корзина, checkout, заказы, аккаунт, админка, API.
2. **Theme layer** — токены цветов/типографики, логотипы, баннеры, контентные блоки.
3. **Module layer** — оплаты, доставка, маркетинг-инструменты, интеграции.
4. **Tenant config** — домен, реквизиты, контакты, налоги, локальные настройки.

Базовый принцип: клиентские проекты не форкают код, а настраиваются через тему + конфиг + подключенные модули.

## Быстрый деплой для клиента (операционная модель)

Предусматриваем формат “быстрого развертывания”:

- `docker compose` как базовый runtime (dev/stage/prod максимально похожи);
- `.env` + `tenant.yaml/json` как основной слой настройки;
- команда bootstrap для запуска клиента (миграции, сид базовых справочников, smoke-check);
- шаблон Nginx + HTTPS + резервные копии БД;
- CI пайплайн: тесты -> сборка -> деплой;

## Git и CI/CD (базовая схема)

Чтобы дальше работать спокойно и без хаоса, фиксируем базовый процесс:

- Основная ветка проекта: `main`
- Локальная разработка идет в отдельных рабочих ветках:
  - `feature/<short-name>`
  - `fix/<short-name>`
  - `chore/<short-name>`
- В `main` попадает только то, что прошло локальную проверку и готово к публикации.
- Базовый CI уже подключен через GitHub Actions:
  - backend: `composer install` + `php artisan test`
  - frontend: `npm ci` + `npm run build`
  - workflow запускается на `push` и `pull_request` в `main`
- Пока деплой не автоматизируем: сначала CI проверяет качество, а сам релиз на сервер будем включать отдельным следующим шагом.

Практический режим работы:

1. Работаем локально.
2. Делаем коммиты в git.
3. Пушим изменения в GitHub.
4. GitHub Actions автоматически гоняет проверки.
5. После этого уже отдельно настраиваем staging/production deploy.
- чеклист handover (доступы, webhooks, SMTP, домен, мониторинг).

## Прод-деплой на сервер (обновлено: 24 апреля 2026)

Минимум, что должно быть на сервере:

- Ubuntu 22.04/24.04 LTS
- Docker Engine + Docker Compose v2
- Git
- открытые порты `22`, `80` (и позже `443` под SSL)

Рабочая схема деплоя (проверена):

1. `git clone` проекта в `/opt/shoria`.
2. Заполнить корневой `.env` и `backend/.env` под production (`APP_ENV=production`, `APP_DEBUG=false`, корректные `APP_URL/FRONTEND_URL`).
3. Поднять контейнеры:
   `docker compose up -d --build`.
4. Инициализировать backend:
   `docker compose exec app composer install --no-dev --optimize-autoloader --no-interaction`
   `docker compose exec app php artisan key:generate --force`
   `docker compose exec app php artisan migrate --force`
   `docker compose exec app php artisan db:seed --class=ShopDemoSeeder --force`
   `docker compose exec app php artisan storage:link`
5. Выдать права на каталоги Laravel:
   `docker compose exec app chown -R www-data:www-data /var/www/html/storage /var/www/html/bootstrap/cache`
   `docker compose exec app chmod -R ug+rwX /var/www/html/storage /var/www/html/bootstrap/cache`
6. Проверить доступность:
   `curl -I http://127.0.0.1/`
   `curl -I http://127.0.0.1/admin/login`
   `curl -I http://127.0.0.1/api/categories`

Если при сборке встретили Docker Hub rate limit (`toomanyrequests`), временно используйте зеркала `public.ecr.aws/docker/library/*` для базовых образов (`php`, `nginx`, `postgres`, `redis`, `composer`, `node`) в `docker-compose.yml` и `docker/php/Dockerfile`.

Важно по безопасности:

- после первичной настройки обязательно сменить root-пароль;
- перейти на SSH-ключи и отключить вход по паролю;
- включить HTTPS после привязки домена.

## Платежи РФ: модульный контур (эквайринг/банки)

Чтобы безболезненно подключать разные банки у разных клиентов, делаем **платежный слой через адаптеры**:

- `PaymentGatewayInterface` (единый контракт): `createPayment`, `getStatus`, `refund`, `handleWebhook`.
- `PaymentProviderRegistry` (реестр доступных провайдеров).
- отдельные модули-провайдеры:
  - `TBankAcquiringAdapter`
  - `YooKassaAdapter`
  - `CloudPaymentsAdapter`
  - `SberAdapter` (этапно, по доступному API-сценарию)
- хранение ключей/секретов только в env/secret-store;
- идемпотентность, подписи webhook, журнал транзакций, retry-стратегия.

Что важно сразу предусмотреть в доменной модели:

- таблицы `payment_transactions`, `payment_webhook_logs`, `payment_refunds`;
- привязка транзакции к `order_id` + внешнему `provider_payment_id`;
- единый статус-мэппинг провайдера -> внутренний статус заказа.

### Целевая операционная модель заказа

До подключения реального эквайринга фиксируем целевую модель так, чтобы потом не переделывать `orders` задним числом.

- У заказа не должно быть только одного поля `status`.
- Нужны независимые контуры:
  - `order_status` — общий бизнес-статус заказа
  - `payment_status` — состояние денег
  - `fulfillment_status` — сборка/отгрузка/выдача
  - `refund_status` — возвраты

Базовый набор статусов:

- `order_status`:
  - `placed` — заказ оформлен
  - `confirmed` — подтвержден магазином
  - `completed` — успешно завершен
  - `cancelled` — отменен
- `payment_status`:
  - `unpaid` — платеж не начат
  - `pending` — платеж создан, ждем ответ банка/webhook
  - `authorized` — деньги захолдированы
  - `paid` — деньги подтверждены банком
  - `failed` — платеж неуспешен
  - `cancelled` — платеж отменен
  - `partially_refunded` — частичный возврат
  - `refunded` — полный возврат
- `fulfillment_status`:
  - `pending` — еще не обрабатывается
  - `processing` — собирается
  - `packed` — упакован
  - `shipped` — передан в доставку
  - `ready_for_pickup` — готов к выдаче
  - `delivered` — доставлен/выдан
  - `returned` — вернулся обратно
- `refund_status`:
  - `none`
  - `requested`
  - `approved`
  - `processing`
  - `partially_refunded`
  - `refunded`
  - `failed`

Правила переходов:

- После checkout:
  - `order_status=placed`
  - `payment_status=unpaid|pending`
  - `fulfillment_status=pending`
- После подтверждения оплаты банком:
  - `payment_status=paid`
  - заказ можно переводить в `confirmed`
- После начала сборки:
  - `fulfillment_status=processing`
- После передачи в доставку:
  - `fulfillment_status=shipped`
- После успешной выдачи/доставки:
  - `fulfillment_status=delivered`
  - `order_status=completed`
- При отмене:
  - если оплаты не было: `order_status=cancelled`, `payment_status=cancelled|failed`
  - если оплата была: сначала создаем возврат, потом переводим заказ в `cancelled`

Возвраты и чеки:

- Возврат нельзя хранить только флагом в заказе, нужна отдельная таблица `payment_refunds`.
- Для каждого возврата храним:
  - `order_id`
  - `payment_transaction_id`
  - `amount`
  - `reason`
  - `status`
  - `provider_refund_id`
  - `requested_by`
  - `processed_at`
- Для фискализации нужен отдельный контур чеков:
  - чек продажи
  - чек возврата прихода
  - при частичном возврате — отдельный частичный возвратный чек

Доставка:

- У заказа/отправления должны появиться служебные поля:
  - `carrier_code`
  - `tracking_number`
  - `tracking_url`
  - `delivery_service_status`
  - `shipped_at`
  - `delivered_at`
- Для `pickup` нужен отдельный сценарий `ready_for_pickup`.
- Для курьерской доставки нужен сценарий `shipped -> delivered` или `shipped -> returned`.

Что заложить в схему данных перед эквайрингом:

- В `orders`:
  - `order_status`
  - `payment_status`
  - `fulfillment_status`
  - `refund_status`
  - `confirmed_at`
  - `cancelled_at`
  - `completed_at`
- Таблицы:
  - `payment_transactions`
  - `payment_webhook_logs`
  - `payment_refunds`
  - `order_status_logs`
- Опционально:
  - `delivery_events` или `shipment_events`

Что нужно в админке:

- Отдельные блоки:
  - `Оплата`
  - `Исполнение`
  - `Возвраты`
  - `История статусов`
- Ручные действия:
  - `Подтвердить заказ`
  - `Отметить как оплаченный` только для offline/manual-сценариев
  - `Передать в сборку`
  - `Передать в доставку`
  - `Отметить как доставленный`
  - `Отменить заказ`
  - `Создать возврат`
- Все ручные изменения должны логироваться с указанием оператора.

Минимальный безопасный live-процесс для первой версии:

- Создаем заказ в `placed`.
- Создаем `payment_transaction`.
- Ждем webhook банка.
- Только webhook банка переводит оплату в `paid`.

## Журнал решений

- Для админки выбрали **Filament v4** (совместим с Laravel 13).
- Публичная витрина остается на **Vue SPA**, бэкенд отдает JSON API.
- Для первого этапа храним изображения как `image_url` (быстро поднять MVP).
- Демо-контент сидится автоматически через `DatabaseSeeder -> ShopDemoSeeder`.
- Аналитические события на раннем этапе пишем в `tracking_events`, чтобы не потерять данные MVP.
- Каталог пока работает от server-side пагинации (`/api/products`) без клиентского кеша.
- Для ЛК покупателей выбрали гибридную модель идентичности: guest (`session_id`) + user (`Sanctum` token).
- После checkout делаем redirect на страницу успешного заказа, данные подтягиваются из API заказа; история заказов вынесена на отдельную страницу.
- Поток восстановления пароля сделали API-first (подходит и для SPA, и для будущего мобильного клиента).
- Email verification также сделали API-first: ссылка подтверждения подписанная (`signed URL`), статус доступен в `/api/auth/me`.
- Для UX email verification добавлен backend->frontend redirect, чтобы пользователь не оставался на техническом JSON endpoint.
- Для размеров сделали variant-first модель: выбранный размер фиксируется в `cart_items` и переносится в `order_items` как snapshot-поле `variant_label`.
- Для checkout v2 выбрали серверный расчет totals (скидка/доставка), чтобы фронт не мог исказить финальную сумму.
- Доставка и промокоды вынесли в управляемые сущности (`delivery_methods`, `promo_codes`) и подключили к checkout через API.
- Для UX промокодов добавили отдельный preview-endpoint, чтобы показывать пользователю точный server-side статус промокода до submit.
- Для ограничения злоупотреблений ввели отдельный журнал `promo_code_usages` и проверку одноразового использования по email.
- Для real-world эквайринга заложили шаблонный слой интеграций: `payment_providers`, `delivery_providers`, `payment_transactions`, registry/driver-архитектуру и test/live режимы.
- В checkout публичный `payment_method` оставили бизнес-типом (`card` / `cash`) для совместимости UI, а конкретный банк/провайдер фиксируется в `payment_transactions.provider`.
- Для шаблонного продукта выбрали сценарий "настройка через админку": владелец магазина заполняет доступы провайдера в Filament, а активные способы оплаты/доставки автоматически попадают в checkout.
- Демо-проект по умолчанию стартует с `T-Bank (sandbox)` + `Наличными при получении`, а для доставки уже заложены `Локальная доставка` и отключенный `CDEK (sandbox)`.

## Перенос «с локалки на сервер»

Ничего специального ломать не нужно: образы и переменные окружения те же; на проде задаёте секреты, домен, `APP_ENV=production`, отключаете лишнее, подключаете бэкапы. SPA собирается в статику (`npm run build`) и отдаётся Nginx или CDN; API остаётся за тем же Nginx + PHP‑FPM.
